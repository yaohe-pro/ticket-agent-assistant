# 工单 Agent 记忆、状态与人工确认阶段总结

更新时间：2026-06-19

## 这一阶段完成了什么

这一阶段把 `ticket-agent-assistant` 从“能处理单个工单的 Agent”升级成了“能参考历史、识别重复、人工确认、保存审计日志、运行评估”的项目雏形。

也就是说，它不再只是一个会回答的脚本，而开始具备真实 Agent 应用需要的几个关键能力：

- 记住以前处理过什么
- 判断当前工单是不是重复工单
- 在高风险或重复场景下让人确认
- 根据历史案例辅助生成回复
- 把处理过程保存下来，方便复盘和评估
- 用评估脚本证明核心规则可用

## 本阶段目标

1. 让 Agent 读取同一工单的历史处理记录。
2. 让 Agent 读取相似历史工单，作为生成回复草稿的上下文。
3. 把重复工单确认前置，避免用户选择停止时还继续调用 RAG 和大模型。
4. 把重复工单判断规则拆到单独策略模块，方便测试和维护。
5. 增加日志分析、记忆检查和一键评估脚本，让项目更像可展示的工程项目。

## 核心概念

### 1. State

State 就是 Agent 在流程中一直携带的“工作单”。每个节点处理完后，都会往 State 里补充信息。

在本项目里，State 里保存了：

- 工单编号
- 工单内容
- 分类和优先级
- 是否使用知识库
- 同一工单历史次数
- 是否继续处理重复工单
- 相似历史工单
- 回复草稿
- 人工审核结果
- 最终状态

### 2. 记忆

这里的记忆不是模型自己记住，而是我们把历史处理日志存成 JSON，再在处理新工单时读取出来。

当前实现了两种记忆：

- 同一工单历史：判断这个工单以前是否处理过。
- 相似历史工单：找到类别、优先级和关键词相近的历史案例，给回复草稿提供参考。

### 3. Human-in-the-loop

Human-in-the-loop 指关键步骤让人参与确认。

本项目里有两类人工确认：

- 重复工单确认：如果工单以前处理过，先问人是否继续。
- 回复发送确认：草稿生成后，问人是否发送给用户。

这体现了生产 Agent 的一个重要原则：模型可以辅助决策，但高风险动作要有人把关。

### 4. 策略模块

重复工单是否需要确认，本质是业务规则。我们把它放进 `policy.py`，而不是散落在 Agent 主流程里。

这样做的好处是：

- 主流程更清晰
- 规则可以单独测试
- 以后修改策略不用大改 Agent 图

### 5. 评估

评估是为了证明项目不是“刚好跑通一次”，而是核心能力可以重复验证。

当前有三类评估：

- 分类和优先级评估
- 重复工单策略评估
- 历史记忆检索评估

`run_all_evals.py` 会一次运行全部评估，并生成总览报告。

## 涉及文件

| 文件 | 作用 |
| --- | --- |
| `ticket_agent/agent.py` | LangGraph 主流程，包含读取工单、分类、重复检查、历史记忆、RAG、草稿生成、人工审核和日志保存 |
| `ticket_agent/storage.py` | 处理日志读写、同一工单历史查询、相似历史工单检索 |
| `ticket_agent/policy.py` | 重复工单策略判断 |
| `ticket_agent/classifier.py` | 工单分类和优先级规则 |
| `ticket_agent/rag.py` | 知识类工单的查询改写和企业知识库检索 |
| `main.py` | 命令行运行入口 |
| `analyze_logs.py` | 分析历史处理日志 |
| `inspect_memory.py` | 查看某个工单会匹配哪些历史记忆 |
| `eval_classification.py` | 分类和优先级评估 |
| `eval_duplicate_policy.py` | 重复工单策略评估 |
| `eval_memory_retrieval.py` | 历史记忆检索评估 |
| `run_all_evals.py` | 一键运行全部评估 |
| `data/ticket_process_log.json` | 工单处理历史日志 |
| `data/evaluation_summary.json` | 一键评估总览报告 |

## 你应该掌握什么

完成这一阶段后，你要能用自己的话讲清楚：

1. LangGraph 的 State 为什么重要。
2. 为什么 Agent 不能只靠一次 LLM 调用。
3. 日志为什么可以变成长期记忆。
4. 同一工单历史和相似历史工单有什么区别。
5. 为什么重复工单确认要放在 RAG 和 LLM 调用之前。
6. 为什么人工确认是企业 Agent 的关键能力。
7. 为什么要把业务规则拆成 `policy.py`。
8. 为什么项目需要评估脚本，而不是只手动试一次。

## 面试可以这样讲

这个项目里，我用 LangGraph 把企业工单处理拆成多个节点：读取工单、分类、重复工单检查、历史记忆检索、知识库 RAG 检索、回复草稿生成、人工确认和日志保存。

为了让 Agent 更接近真实企业场景，我加入了同一工单历史检测和相似历史工单记忆。系统发现重复工单时，会在调用 RAG 和大模型之前要求人工确认，避免重复发送和不必要的模型成本。

项目还提供了日志分析、记忆检查工具和一键评估脚本，用来验证分类、重复策略和历史记忆检索效果。

## 当前不足

- 历史记忆还是关键词打分，不是向量语义检索。
- 工单数据仍是模拟数据。
- 当前主要是命令行交互，还没有网页界面。
- 评估用例数量还比较少。
- 还没有接入真实工单系统或消息发送接口。
