# 企业工单处理 Agent 项目复盘

## 项目定位

这是一个面向企业工单处理场景的 Agent 应用雏形，用来展示 LangGraph 编排、LLM 生成草稿、人工审核、日志记录和评估报告这些 Agent 应用开发核心能力。

项目不是简单聊天机器人，而是一个有明确业务流程的任务处理系统。

## 解决的问题

企业客服或内部支持团队每天会收到很多工单，例如：

- 用户找不到知识库答案
- 用户询问产品或适用人群
- 系统打不开影响培训

这个项目模拟一个 Agent 帮助团队完成：

```text
读取工单
-> 分类和判断优先级
-> 读取同一工单历史
-> 如果是重复工单，先确认是否继续处理
-> 如果选择不继续，直接保存日志并停止
-> 读取相似历史工单作为上下文记忆
-> 知识类工单改写查询并检索企业知识库
-> 生成回复草稿
-> 人工审核
-> 发送或等待修改
-> 保存处理日志
-> 统计和评估处理结果
```

## 当前功能

- 读取模拟工单数据
- 判断工单类别和优先级
- 对知识类工单进行查询改写
- 检索企业知识库 RAG 资料
- 从历史处理日志中读取同一工单过去的处理记录
- 在调用 RAG 和 LLM 之前进行重复工单二次确认
- 从历史处理日志中读取相似工单，作为回复草稿的上下文记忆
- 调用 DeepSeek 生成回复草稿
- 在回复发送前进行人工确认
- 提供 Streamlit 网页页面，支持选择工单、生成草稿、查看历史记忆并人工审核
- 人工通过后发送，拒绝后等待人工修改
- 保存工单处理日志到 JSON
- 分析历史处理日志，包括审核结果、工单类别、优先级和知识库使用情况
- 评估分类和优先级规则
- 导出分类评估报告
- 记录每次使用了哪些历史工单作为记忆来源

## 技术架构

```text
main.py
  -> ticket_agent.agent.process_ticket()
      -> load_ticket_node
      -> 同一工单历史在 load_ticket_node 中读取
      -> classify_ticket_node
      -> retrieve_history_node
      -> retrieve_knowledge_node
      -> draft_reply_with_llm_node
      -> human_review_node
      -> send_reply_node / stop_reply_node
      -> save_process_log_node

analyze_logs.py
  -> 读取 ticket_process_log.json
  -> 统计审核结果、类别、优先级和最终状态

eval_classification.py
  -> 运行固定评估用例
  -> 计算分类准确率
  -> 保存 classification_eval_report.json

eval_duplicate_policy.py
  -> 运行固定重复工单策略用例
  -> 验证同一工单历史次数是否触发人工确认
  -> 保存 duplicate_policy_eval_report.json

eval_memory_retrieval.py
  -> 运行固定记忆检索用例
  -> 检查相似历史工单是否按类别和优先级召回
  -> 保存 memory_retrieval_eval_report.json

run_all_evals.py
  -> 依次运行分类评估、重复工单策略评估和记忆检索评估
  -> 汇总每个评估项的测试数量、正确数量和准确率
  -> 保存 evaluation_summary.json

inspect_memory.py
  -> 输入工单编号
  -> 展示分类结果、相似历史、相似分数和匹配关键词
  -> 用来调试 Agent 记忆是否合理

streamlit_app.py
  -> 选择工单
  -> 生成处理草稿
  -> 展示重复工单提醒、历史记忆、RAG 资料和回复草稿
  -> 人工点击按钮审核
  -> 查看日志统计和最近处理记录
```

## 核心设计

### 1. 用 LangGraph 管理流程

项目没有把所有逻辑写在一个函数里，而是拆成多个节点：

- 读取工单节点
- 分类节点
- 回复草稿节点
- 人工审核节点
- 发送或取消节点
- 日志保存节点

这样做的好处是流程更清晰，也方便后续插入知识库查询、人工确认、失败重试和评估节点。

### 2. LLM 只负责生成草稿

DeepSeek 用来生成自然语言回复草稿，但不直接决定是否发送。

这样可以减少模型误操作风险。

### 3. 知识类工单先查企业知识库

知识类工单不会直接让模型凭空回复，而是先调用企业知识库 RAG。

工单原文常常是投诉式表达，例如：

```text
知识库问答没有找到产品介绍
```

这种文本不适合直接检索，因此项目加入了 query rewrite，把它改写成：

```text
蓝海科技有哪些主要产品？
```

再进行混合检索和重排序。这样回复草稿可以直接包含知识库中的产品信息，而不是只说“我们会处理”。

### 4. 高风险动作需要人工确认

回复用户属于业务动作，所以项目加入了人工审核节点：

```text
yes -> 发送回复
no -> 等待人工修改
```

这体现了 human-in-the-loop 的设计思想。

如果同一工单之前已经处理过，项目会在 RAG 检索和 LLM 生成之前先提醒用户，并询问是否仍然继续处理。这可以降低重复发送或误处理的风险，也能避免无意义的大模型调用成本。

### 5. 处理过程可追踪

每次处理工单都会保存：

- 工单编号
- 工单内容
- 类别
- 优先级
- 是否使用企业知识库资料
- 同一工单之前被处理过几次
- 重复工单提醒后的人工选择
- 是否使用历史记忆
- 参考过的历史工单编号
- 回复草稿
- 人工审核结果
- 最终状态

这些日志可用于审计、排错、评估和面试展示。

### 6. 历史日志可以作为长期记忆

项目现在不只是把日志保存下来，还会在处理新工单时读取同类别、同优先级且关键词更相似的历史工单。

这一步的意义是：

```text
过去处理过什么
-> 找出同一工单历史和相似案例
-> 作为新工单生成回复时的参考上下文
```

例如再次处理产品类知识工单时，Agent 会优先参考包含“产品、介绍、知识库”等关键词的历史工单；处理系统打不开问题时，会优先参考包含“打不开、系统、页面、培训”等关键词的历史工单。但最终事实仍然以当前知识库资料为准。

如果同一个工单编号之前已经处理过，Agent 也会把最近几次处理记录放进上下文，让它知道这不是第一次处理。

### 7. 有最小评估闭环

`eval_classification.py` 会评估分类和优先级是否符合预期，并生成 JSON 报告。

`eval_duplicate_policy.py` 会评估重复工单策略是否符合预期，例如 0 次历史不需要确认，1 次或多次历史需要确认。

`eval_memory_retrieval.py` 会评估历史记忆检索是否能找到预期类别、优先级和关键词相关的历史工单。

`run_all_evals.py` 可以一键运行当前全部评估，并生成总览报告。

当前评估结果：

```text
评估总数：3
正确数量：3
准确率：1.0
```

历史记忆检索当前评估结果：

```text
评估总数：2
正确数量：2
准确率：1.0
```

重复工单策略当前评估结果：

```text
评估总数：3
正确数量：3
准确率：1.0
```

全部评估总览当前结果：

```text
评估项数量：3
是否全部通过：True
```

## 面试可讲点

### 为什么用 LangGraph？

因为工单处理不是一次模型调用，而是多个步骤组成的流程。LangGraph 可以把流程拆成节点，并用状态在节点之间传递数据。

### 为什么回复草稿要人工审核？

模型生成内容可能不准确或不适合直接发送。人工审核可以降低误发、乱承诺和业务风险。

### 为什么要做查询改写？

工单原文常常是投诉或描述问题，不是标准检索问题。查询改写可以把“知识库没找到产品介绍”改成“蓝海科技有哪些主要产品”，提高检索和回复质量。

### 为什么要保存日志？

日志可以追踪 Agent 做过什么，也可以用于审计、排错和后续评估。

### 历史日志为什么算记忆？

因为 Agent 在处理新工单时会读取过去相似工单，并把这些历史记录放进提示词中。当前版本会按类别、优先级和关键词相似度筛选历史记录，这样日志就从“事后记录”升级成了“可被 Agent 使用的上下文”。

项目还会记录被参考的历史工单编号，这相当于给记忆增加“引用来源”。如果回复质量不好，可以回头检查是不是参考了不合适的历史记录。

同时，项目会统计同一工单之前已经被处理过几次。如果发现重复工单，会在 RAG 和 LLM 调用前要求人工确认是否继续处理。选择停止时，系统会直接保存审计日志。

### 为什么要做分类评估？

评估可以证明规则或模型是否真的符合预期，避免只靠人工感觉判断效果。

### 为什么要做重复工单策略评估？

重复工单是否需要人工确认是业务规则，不应该只靠手动试。把它抽成 `policy.py` 后，可以用 `eval_duplicate_policy.py` 单独验证策略是否正确。

### 为什么要做 run_all_evals.py？

项目评估脚本变多后，如果每次都手动跑多个命令很容易漏。`run_all_evals.py` 把所有评估集中到一个入口，并生成总览报告，方便本地回归测试和面试展示。

### 为什么要做记忆检索评估？

因为“有历史日志”不等于“Agent 用对了历史”。记忆检索评估可以检查新工单是否能找到同类别、同优先级且关键词相关的历史案例，避免把无关历史塞进 prompt。

### 为什么要做 inspect_memory.py？

它是一个调试工具。完整 Agent 会调用大模型，排查起来成本更高；`inspect_memory.py` 只检查记忆匹配过程，可以快速看到相似分数、匹配关键词和历史回复，方便判断记忆是否合理。

## 当前不足

- 工单数据还是写死的模拟数据
- 分类和优先级判断还是规则，不是模型或可配置策略
- 网页页面已有第一版，但还可以继续增强交互和视觉细节
- 暂时没有真实发送接口
- 历史记忆匹配目前是关键词打分，还不是语义相似度检索
- 目前重复工单只做二次确认，还没有更细的重复原因判断
- 评估用例数量较少
