# 工单 Agent Streamlit 产品化总结

更新时间：2026-06-19

## 这一阶段完成了什么

这一阶段把 `ticket-agent-assistant` 从命令行项目升级成了一个可以本地打开的 Streamlit 网页应用。

现在页面可以完成：

- 选择模拟工单
- 生成处理草稿
- 自动检查重复工单
- 展示同一工单历史
- 展示相似历史工单记忆
- 对知识类工单检索企业知识库
- 展示 RAG 查询改写和知识库资料
- 人工审核回复草稿
- 审核通过后保存发送日志
- 审核拒绝后保存取消日志
- 查看最近处理记录和日志统计

## 为什么要做网页产品

命令行脚本可以验证逻辑，但不适合给面试官或非技术用户演示。

网页产品可以更清楚地展示：

- Agent 处理流程
- 哪些步骤是自动完成的
- 哪些步骤需要人工确认
- Agent 使用了哪些知识库资料
- Agent 参考了哪些历史记忆
- 最终日志如何沉淀为可观测记录

这会让项目更像一个真实业务工具，而不只是学习脚本。

## 关键设计

### 1. 不破坏命令行版本

原来的 `main.py` 和 `ticket_agent/agent.py` 仍然保留，继续适合学习 LangGraph 命令行流程。

网页版本新增 `web_workflow.py`，原因是网页里的人工确认不是 `input()`，而是按钮。

### 2. 先暂停，再确认

如果系统发现某个工单之前处理过，页面会先暂停，让你选择：

- 继续处理这个重复工单
- 停止处理并记录日志

这对应企业 Agent 里的风险控制。

### 3. 草稿可编辑后再审核

页面生成回复草稿后，会放到文本框里。你可以先修改，再点击：

- 审核通过并发送
- 不发送，等待人工修改

这比命令行更接近真实客服审核流程。

### 4. 观测能力前置

页面不只展示结果，还展示：

- 处理流程节点
- 查询改写
- 知识库资料
- 同一工单历史
- 相似历史工单
- 最近处理日志
- 日志统计

这就是 Agent 项目里很重要的 observability。

## 页面使用步骤

1. 左侧选择一个工单，例如 `T1001`。
2. 点击“生成处理草稿”。
3. 如果出现重复工单提醒，选择继续或停止。
4. 查看处理流程、历史记忆和知识库资料。
5. 检查回复草稿，必要时手动修改。
6. 点击“审核通过并发送”或“不发送，等待人工修改”。
7. 切换到“日志与观测”查看处理统计。

## 你应该掌握什么

完成这一阶段后，你要能讲清楚：

- 为什么命令行 Agent 需要产品化
- 为什么网页里不能直接用 `input()`
- 为什么要单独写 `web_workflow.py`
- Streamlit 页面如何控制 Agent 的暂停和继续
- 人工审核按钮对应 human-in-the-loop
- 日志统计为什么属于可观测性
- 为什么一个简历项目要有可演示页面

## 面试可以这样讲

我把命令行版工单 Agent 产品化成了 Streamlit 网页应用。页面支持选择工单、生成草稿、重复工单风险确认、RAG 资料查看、历史记忆查看、人工审核和日志统计。

因为网页交互和命令行 `input()` 不一样，我没有直接硬改原来的 LangGraph 脚本，而是新增了 `web_workflow.py`，把“生成草稿前的自动流程”和“人工审核后的保存流程”拆开。这样既保留了学习用的命令行版本，也让项目具备可演示的产品形态。
